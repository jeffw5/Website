PHASE 0 Concept Stage
Pre-Milestone A
Mission analysis · stakeholder needs elicitation · concept of operations · feasibility · technology landscape assessment. Purpose: establish that a real capability gap exists and that it is worth investing in technology maturation to address it.

CON-001
Concept of Operations (ConOps)
Describes the mission environment, operational scenarios, user roles, and high-level system behavior from the operator's perspective. Not a technical specification — a narrative of intended use.
 Business / Mission Analysis MCD RDSG governed MS A
CON-002
Mission Concept Description (MCD)
Formal statement of the operational mission the system must enable. Captures mission threads, operational context, and the primary capability gap being addressed. Foundation of everything that follows.
 Business / Mission Analysis MCD RDSG governed MS A
CON-003
Capability Concept Description (CCD)
Translates the mission need into a set of required capabilities — what the system must be able to do, without specifying how. Maps capabilities to mission threads from the MCD.
 Stakeholder Needs & Requirements Definition CCD RDSG governed MS A
CON-004
Initial Capabilities Document (ICD)
Joint Staff / service document formalizing the capability gap and need. Precedes and informs the MCD. Establishes the requirement for a materiel solution.
 Business / Mission Analysis MCD reference MS A
CON-005
Technology Assessment Report
Survey of technology readiness levels across candidate solution approaches. Identifies which technologies need maturation and at what cost and risk before commitment is defensible.
 Business / Mission Analysis — input MS A
CON-006
Analysis of Alternatives (AoA)
Systematic comparison of alternative approaches to satisfying the capability need. Evaluates cost, schedule, performance, and risk tradeoffs across materiel and non-materiel options.
 System Analysis — input MS A
Milestone A — Program Initiation
Decision to enter Technology Maturation & Risk Reduction. MCD and CCD must be approved. AoA complete.
PHASE 1 Technology Maturation & Risk Reduction (TMRR)
MS A → Milestone B
The race John describes: get the most abstract and foundational system definition artifacts written, technically reviewed, and governance-verified before Milestone B. No detailed design. No hardware. No code. The V-model left downstroke stops at Architecture Definition. This is where RDSG and SDD L1–L2 provide continuous info-data governance without requiring any design to exist.
ISO 15288 processes active: Stakeholder Needs & Requirements Definition · System Requirements Definition · Architecture Definition · System Analysis
ID Artifact / Info-Data Item ISO 15288 Process John's Category RDSG / SDD Required for
TMR-001
Stakeholder Requirements Specification (StRS)
Formally captures what stakeholders — operators, maintainers, acquirers — need the system to accomplish. Written in stakeholder language, not engineering language. The bridge between the MCD/CCD and the System Requirements Specification.
 Stakeholder Needs & Requirements Definition CCD RDSG primary MS B
TMR-002
System Requirements Specification (SyRS)
The definitive technical statement of what the system must do, expressed as verifiable requirements. Each requirement must be traceable to a stakeholder need and must have a defined acceptance criterion. The RDSG's SHACL layer enforces structural and semantic integrity on this document continuously.
 System Requirements Definition CCD RDSG primary MS B
TMR-003
Functional Concept Description (FCD)
Defines the functional behavior of the system — what functions it performs, in what sequence, with what inputs and outputs — without specifying which physical elements perform each function. The functional architecture layer. Answers "what does it do" not "what is it made of."
 Architecture Definition FCD RDSG primary MS B
TMR-004
Architecture Concept Description (ACD)
Defines the system's architectural structure — the physical and logical elements, their boundaries, relationships, and interfaces — at the conceptual level. Allocates functions from the FCD to architectural elements. The most concrete artifact appropriate before MS B. Becomes the primary input to post-MS B design definition.
 Architecture Definition ACD RDSG primary MS B
TMR-005
Interface Requirements Specification (IRS)
Defines the external and internal interface requirements at the architectural boundary level — data rates, protocols, timing constraints, electrical characteristics. Not yet an Interface Control Document (which specifies how interfaces are implemented — that is EMD work).
 Architecture Definition ACD RDSG governed MS B
TMR-006
System Engineering Plan (SEP)
Defines how the program will execute systems engineering across the lifecycle — processes, methods, tools, decision gates, workforce structure, data rights strategy, STPA integration plan. The governance document for all technical work. Required before MS B.
 Technical Management — RDSG referenced MS B
TMR-007
Technology Readiness Assessment (TRA)
Independent assessment of Technology Readiness Levels for all critical technologies. Must demonstrate TRL ≥ 6 for key technologies before MS B. GAO consistently cites immature technologies at commitment as a primary MDAP failure driver.
 System Analysis — input MS B
TMR-008
STPA Loss Scenario Report
Output of System-Theoretic Process Analysis applied to the functional architecture. Identifies unsafe control actions — the conditions under which system functions produce hazardous outcomes. Defines the UCA boundary set that L1 Scope Drift Detection monitors against during EMD operations.
 System Analysis (Safety) FCD / ACD RDSG + SDD L1 MS B
TMR-009
MOVOI Decision Analysis Report
Multiobjective Value of Information analysis quantifying which design alternatives retain epistemic justification for continued consideration, and which should be eliminated from the set. The formal knowledge-gate mechanism of Set-Based Design. Updated as technology data matures through TMRR.
 System Analysis — SDD L4 trigger MS B
TMR-010
Risk Management Plan & Risk Register
Identifies, assesses, and tracks technical risks across cost, schedule, performance, and integration dimensions. Feeds the Independent Technical Risk Assessment at MS B.
 Risk Management — input MS B
TMR-011
Independent Technical Risk Assessment (ITRA)
Required by DoDI 5000.88 before MS B. Independent review team evaluates the maturity and soundness of the technical baseline, identifies residual risks, and provides the Milestone Decision Authority with an independent view of program readiness. Separate from the program office's self-assessment.
 Technical Review — evidence MS B
TMR-012
SysML v2 Repository Baseline (KerML core)
The machine-readable, queryable representation of the TMR-001 through TMR-008 info-data items maintained in the RDSG digital thread. Not a separate document — the semantic encoding of all the above artifacts. The SHACL shape constraints validate structural integrity continuously.
 Architecture Definition ACD foundation RDSG IS this MS B
Milestone B — Development Start
The knowledge commitment gate. SyRS, FCD, and ACD must be complete, verified, and stable. All MS B ITRA criteria satisfied. After this point: design definition begins, EMD contract awarded, detailed descriptions authorized.
PHASE 2 Engineering & Manufacturing Development (EMD) — Pre-CDR
MS B → CDR
Design definition begins. The Architecture Concept Description is the primary input. Physical, hardware, software, database, and algorithm descriptions are produced and progressively detailed. Virtual implementations (Model-in-the-Loop, co-simulation) are appropriate here — physical hardware commitment waits for CDR confidence. This is where the V-model prototype roadmap Phases 2 and 3 properly belong.
ISO 15288 processes active: Design Definition · Implementation · Integration · Verification (component/unit level) · System Analysis (ongoing)
ID Artifact / Info-Data Item ISO 15288 Process John's Category RDSG / SDD Required for
EMD-001
Physical / Product Concept Description (PCD)
Top-level design description mapping the ACD's architectural elements to physical implementation concepts. The first design artifact — bridges architecture to detailed design without yet specifying manufacturing or fabrication details. Takes the ACD as direct input.
 Design Definition PCD RDSG + SDD L2 PDR
EMD-002
Hardware Requirements Specification (HRS)
Allocates system requirements to hardware elements. Specifies electrical, mechanical, thermal, and environmental requirements for each hardware configuration item. Traceable to the SyRS.
 Design Definition PCD SDD L2 PDR
EMD-003
Software Requirements Specification (SRS)
Allocates system requirements to software configuration items. Defines software functions, performance constraints, interface requirements, and quality attributes. Traceable to the SyRS and FCD.
 Design Definition PCD SDD L2 PDR
EMD-004
Interface Control Document (ICD)
Specifies how interfaces defined in the IRS are actually implemented — signal definitions, data formats, timing, electrical characteristics, connector pinouts. The definitive bilateral agreement between interfacing elements. Required before integration can proceed.
 Design Definition PCD SDD L1+L2 PDR
EMD-005
Algorithm Specification Document
Specifies AI, control, signal processing, and decision algorithms at the mathematical level — inputs, outputs, computational method, performance bounds, training data provenance (for AI), and uncertainty representation. MTBH and Brier Score thresholds are defined here.
 Design Definition PCD SDD L3 baseline PDR
EMD-006
Database Design Description (DDD)
Specifies the logical and physical structure of all databases — schema, data dictionaries, retention policies, access controls. Includes ontology specifications for semantic data stores supporting the RDSG.
 Design Definition PCD SDD L2 PDR
EMD-007
Hardware Design Description (HDD)
Detailed design documentation for hardware configuration items — circuit schematics, PCB layouts, mechanical drawings, thermal analyses. The level of detail that makes procurement and fabrication possible.
 Design Definition PCD SDD L1 CDR
EMD-008
Software Design Description (SDD-doc)
Detailed design documentation for software configuration items — architectural decomposition, module designs, class diagrams, data flow. Note: distinct from the Scope Drift Detection (SDD) framework; context differentiates.
 Design Definition PCD SDD L2+L3 CDR
EMD-009
Virtual Integrated System (co-simulation model)
Functional Mockup Unit assembly coupling physics models (Modelica) and control logic (Simulink) via FMI 3.0. The Model-in-the-Loop implementation that exercises the PCD and HRS/SRS without physical hardware. Appropriate pre-CDR; used to generate PDR-level evidence.
 Implementation · Integration realized product SDD L1+L2 PDR
EMD-010
Test & Evaluation Master Plan (TEMP)
Defines the overall developmental and operational test strategy — test events, test articles, success criteria, instrumentation. Links each requirement in the SyRS to a specific test method and acceptance criterion.
 Verification — evidence PDR
EMD-011
Preliminary Design Review (PDR) Package
Technical review artifact set demonstrating that the system design is sufficiently mature to proceed to detailed design. Includes design descriptions at PCD level, interface status, preliminary TEMP, risk status, and mass / power / performance budgets.
 Technical Review — evidence gate PDR
EMD-012
Drawings, Parts Lists, Bills of Materials
Engineering release-level documentation specifying exact materials, dimensions, tolerances, and assembly instructions. The artifacts that make physical procurement and fabrication possible. These do NOT exist before CDR — their premature creation is the root cause of premature lock-in that John's feedback addresses.
 Design Definition PCD SDD L1 CDR
Critical Design Review (CDR)
Confirms detailed design is stable enough to begin physical procurement and assembly. All design descriptions complete. HiL integration readiness demonstrated. After CDR: real hardware is procured and physical system realization begins.
PHASE 3 EMD Post-CDR — Integration, Verification & Validation
CDR → Milestone C
The right upstroke of the V-model. Physical system is built and verified against design descriptions, then validated against stakeholder requirements and mission needs. Hardware-in-the-Loop integration operates here. Scope Drift Detection L1 telemetry is at full operational fidelity. MTBH and Brier Score monitoring is against physical system behavior, not simulation.

Milestone C — Production Decision
Authorizes low-rate initial production. System verification and validation complete. Production drawings and specifications released.
PHASE 4 Production, Operations & Support
Post-MS C
System fielded and operational. Scope Drift Detection shifts from development monitoring to operational monitoring — watching for emergent behaviors in deployed AI components, tracking whether operational conditions match the envelope the system was designed and validated for.

Terminology reference

Plain-English definitions for shared vocabulary
MCD — Mission Concept Description
Formal statement of the operational mission the system must enable. Captures mission threads and the primary capability gap.
CCD — Capability Concept Description
Translates the mission need into required capabilities — what the system must be able to do, without specifying how.
FCD — Functional Concept Description
Defines system functional behavior — what functions it performs, in what sequence — without specifying which physical elements perform them.
ACD — Architecture Concept Description
Defines system architectural structure — elements, boundaries, relationships, interfaces — at the conceptual level. The most concrete pre-MS B artifact.
PCD — Physical / Product Concept Description
Post-MS B design description mapping architectural elements to physical implementation concepts. First design artifact; takes ACD as input.
SyRS — System Requirements Specification
Definitive technical statement of what the system must do, with verifiable requirements traceable to stakeholder needs.
IRS — Interface Requirements Specification
Architectural-level interface requirements. Precedes the ICD (which specifies implementation) — an IRS defines what interfaces must do; an ICD specifies how they do it.
ICD — Interface Control Document
Bilateral agreement specifying how an interface between two elements is actually implemented. An EMD artifact, not a TMRR artifact.
STPA — System-Theoretic Process Analysis
Safety analysis method identifying unsafe control actions — ways system functions can produce hazardous outcomes even when individual components function correctly.
UCA — Unsafe Control Action
An STPA-identified condition under which a system control action is hazardous — applied too early, too late, for too long, or not at all in a given context.
MOVOI — Multiobjective Value of Information
Analysis quantifying when a program has accumulated sufficient knowledge to justify committing to a design alternative — the formal knowledge gate of Set-Based Design.
RDSG — Requirements-Driven Semantic Gateway
Machine-readable, machine-enforceable governance infrastructure for the info-data baseline. Uses SHACL constraint validation and SWRL temporal rules to maintain authoritative source-of-truth continuously.
SDD — Scope Drift Detection
Four-layer operational monitoring architecture (L1 telemetry, L2 info-data currency, L3 AI behavioral entropy, L4 knowledge gap surface) that detects divergence between formal baseline and observed system behavior.
MTBH — Mean Time Between Hallucinations
Reliability-theoretic metric for AI component trustworthiness. Declining MTBH is a leading indicator that the system is operating outside its validated operational envelope.
Brier Score
Probabilistic calibration metric for AI uncertainty representation. Rising Brier indicates the AI is becoming overconfident in domains where its reliability is declining.
TMRR — Technology Maturation & Risk Reduction
The acquisition phase between Milestone A and Milestone B. Purpose: prove technologies are mature and risks are understood enough to justify committing to full development.
EMD — Engineering & Manufacturing Development
The phase beginning at Milestone B where detailed design, fabrication, integration, and verification occur. Produces the system that will be evaluated for production at Milestone C.
ITRA — Independent Technical Risk Assessment
Required by DoDI 5000.88 before MS B. Independent review team evaluates technical baseline maturity and provides the Milestone Decision Authority with an assessment separate from the program office's self-report.
SE-V / V-Model
The standard representation of the systems engineering process: left downstroke = system definition (increasingly detailed), bottom = implementation, right upstroke = verification and validation (increasingly abstract).
TRL — Technology Readiness Level
Nine-level scale measuring how mature a technology is, from basic principles observed (TRL 1) to actual system proven in operational environment (TRL 9). MS B typically requires TRL ≥ 6 for critical technologies.
FMI — Functional Mockup Interface
Standard protocol for coupling physics simulation models (e.g., Modelica) with control logic models (e.g., Simulink) in co-simulation. Enables Virtual Integrated System testing without physical hardware.
HiL — Hardware-in-the-Loop
Test configuration where real hardware components are integrated with simulation models representing the rest of the system. Appropriate post-CDR when physical hardware is available and detailed design is stable.
